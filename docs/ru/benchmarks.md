# Бенчмарки

::: info Информация
Лучший способ сравнить библиотеки - выполнить тесты в **вашей собственной среде** с **вашей собственной рабочей нагрузкой**. Каждый проект предъявляет уникальные требования — задержка, пропускная способность, объем памяти, сложность интеграции — и ни один тест не может учесть их все.

Рекомендую вам протестировать `ulog` наряду с другими библиотеками и выбрать инструмент, который наилучшим образом соответствует вашим потребностям.
:::

::: tip Примечание
Используется `WithExtractor("node_id", "trace_id")` для автоматического извлечения из контекста. Все тесты пишут в `io.Discard` (аналог `/dev/null` в Unix или `NUL` в Windows). Измеряются только накладные расходы логирования (форматирование полей, кодирование JSON, извлечение контекста) без учёта дискового или сетевого ввода-вывода. Реальная производительность будет зависеть от назначения вывода (файл, сеть и т.д.).

*Протестировано на Intel Core i9-9880H (2.30 ГГц).*
:::

::: tip Примечание
Используется `WithExtractor("node_id", "trace_id")` для автоматического извлечения из контекста. ## SinkHttp Performance
Данные бенчмарка, измеряющие внутренние затраты HTTP-приёмника `ulog` с использованием `httptest.Server` без сетевой задержки.

### SingleThread
| Mode  | Level                | Operations | Time (ns/op) | Memory (B/op) | Allocs |
|-------|----------------------|------------|--------------|---------------|--------|

| Sync  | **AllSupportLevels** |      13.6K |       82,500 |         9,400 |     85 |

::: tip Примечание
Используется `httptest.Server` для симуляции HTTP-эндпоинта. Измеряются полные накладные расходы: форматирование JSON, извлечение контекста, HTTP-запрос/ответ. В реальной среде задержка в основном определяется сетевым вводом-выводом (обычно в 10–100 раз выше). Эти цифры отражают только внутренние затраты `ulog`. *Multi*-бенчмарки используют `b.RunParallel` для имитации реальной конкурентной нагрузки.

*Протестировано на Intel Core i9-9880H (2.30 ГГц).*
:::
